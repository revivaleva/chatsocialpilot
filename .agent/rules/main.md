# chatsocialpilot Workspace Rules

## 1. 参照優先順位 (Source of Truth)
作業・調査時は以下の順で情報を参照し、事実を把握すること。

1. **CLAUDE.md**: 最新の作業状況、既知のバグ (PITFALL)、完了定義 (DoD)
2. **docs/PROJECT_OVERVIEW.md**: 技術アーキテクチャ、DBスキーマの正解
3. **docs/OPS_DAILY.md**: 運用手順の正解
4. **.cursor/rules/*.mdc**: 補助資料（既存のプロンプト・ルール例として参照）

**重要**
- `.cursor/rules/*.mdc` の内容が `docs/PROJECT_OVERVIEW.md` や実際のコード実装と競合する場合、常に **ドキュメントおよびコード実装を優先** すること。

## 2. 起動・動作確認
- サーバーの起動および UI の動作確認には `npm run dashboard` を使用すること。
- `npm start` は CLI 廃止により意図した動作をしない可能性があるため、dashboard を正とする。

## 3. 壊すとヤバい「境界」 (Critical Boundaries)
- **ID混在**: `container_id` は XID (文字列) と UUID が混在している。DB操作前に必ず対象テーブルの定義を確認する。
- **X投稿欄**: `contenteditable="true"` であるため、通常の `.value` 入力は効かない。
- **排他制御**: 同一コンテナの並列操作（409エラー）を避ける設計を維持する。

## 4. 安全制約と作業原則
- **最小差分**: 無関係な整形・リネーム・移動を厳禁とする。
- **運用優先**: 直接 DB を更新せず、日次運用は `npm run ops:*` コマンド群を介して行う。

- **調査のスクリプト化**: `npx tsx -e` 等の長いワンライナーや、複雑な構造のSQLをターミナルから直接実行することは原則避ける。調査は `scripts/` または `ops/` 配下に再利用可能な `.ts` スクリプトとして作成して実行する。
  - 一度きりの調査でも、複雑なものはスクリプト化して **証跡を残す**（削除は原則しない）。
  - 使い捨てにしたい場合は、削除ではなく `notes/` 配下に移すか、ファイル冒頭に「目的・使用期限」をコメントで残す。

- **コード変更時の検証**: ソースコードを修正した場合は、完了前に `npm run typecheck` を実行し、必要に応じて `npm run lint:fix` と `npm run format` を実行して整合性を確認する。

- **進捗ログ（必須）**: 作業の最後に `notes/ACTIVE_TASK.md` を更新し、目的/現状/実行コマンド/要点/次の一手/変更ファイルを短く記録する（秘匿情報は書かない）。
  - 更新タイミング: コマンド実行後、ファイル変更後、原因仮説確定後、エラー中断時、のいずれかの直後（＝各応答の最後）。
