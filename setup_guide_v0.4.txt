環境構築手順書（v0.4 / Windows 11）

0) ゴール
- 本プロジェクトをWindows 11上で動作させ、Playwrightでブラウザ自動操作し、OpenAI APIへの接続を確認する。
- 仕様書 v0.4 に沿ったディレクトリ初期化・設定ファイル作成・依存導入までを完了。

1) 前提
- Windows 11（管理者権限あり）
- インターネット接続
- OpenAI APIキー（環境変数で設定予定）

2) 推奨バージョン
- Node.js：LTS推奨（v20.x 系）
- npm：同梱版
- Git：最新安定
- Playwright：1.48+（npmインストールで取得）
- SQLite3：同梱バイナリ利用（npmパッケージでOK）

3) 必須インストール（PowerShell）
# Node.js LTS
winget install -e --id OpenJS.NodeJS.LTS

# Git
winget install -e --id Git.Git

# 端末を開き直し、バージョン確認
node -v
npm -v
git --version

4) プロジェクト作成
# 任意の作業フォルダへ
mkdir C:\workspace\social-bot
cd C:\workspace\social-bot

# Node初期化
npm init -y

# TypeScript/ツール群
npm i -D typescript ts-node @types/node eslint prettier

# ランタイム依存
npm i playwright openai better-sqlite3 winston node-cron zod

# TypeScript設定
npx tsc --init

# Playwright ブラウザ取得
npx playwright install

5) フォルダと初期ファイル
# 仕様書に沿って基本構造を作成
mkdir src, src/agent, src/drivers, src/services, src/ui, src/utils, config, storage, storage/snapshots, logs, shots, scripts

# .gitignore（例）
echo node_modules/`n`dist/`n`*.log`n`storage/snapshots/`n`.env > .gitignore

# .env.example
echo OPENAI_API_KEY=sk-xxxx`n`TZ=Asia/Tokyo > .env.example

6) Playwright 確認
# サンプルで起動テスト
npx playwright --version
# 初回はGUI起動のテストも可： npx playwright codegen https://example.com

7) OpenAI 接続確認
# .env を作成し、OPENAI_API_KEY を実キーに変更
copy .env.example .env

8) SQLite 初期化
# better-sqlite3 を使うため、初回実行時に storage/app.db を自動作成する実装（後続の雛形コード）が走る想定

9) Windows タスクスケジューラ（運用移行時）
# MVPはアプリ内スケジューラ（node-cron）で開始
# 常時稼働に移る際は scripts/task-create.xml を用意し、タスク登録

10) Playwright のプロファイル準備
# 既存のElectronコンテナブラウザのユーザーデータディレクトリを流用する場合
# config/accounts.json に userDataDir を設定（例： C:/Profiles/Threads/threadsA）
# ない場合はディレクトリを先に作成し、ログイン自動化（将来実装）でセッションを作る

11) 推奨設定（WAL・Busyリトライ）
- SQLiteはWAL推奨（db.tsで自動化）
- 書き込みは短トランザクション、BUSY時は指数バックオフ（50→100→200ms）

12) 動作テスト（最小）
- npm run build（dist生成） or ts-node 実行
- npx ts-node src/run.ts
- 初回は headless=false で動作確認（後で true に戻す）
- OpenAI呼び出し/投稿ページ遷移/スクショ保存/DBレコード作成を確認

13) トラブルシュート
- Playwrightが起動しない： npx playwright install、GPU/ドライバ、--no-sandbox オプション
- ログイン状態が保持されない： userDataDirの権限/パス確認
- SQLITE_BUSYが多い：並列を下げる・トランザクションを短く・WAL確認
- OpenAIエラー：APIキー/プロキシ/ファイアウォール

14) セキュリティ
- .envは共有しない
- ログ/アーティファクトに機微情報を残さない（マスキング）
- 公式APIがある操作はAPI優先。UI自動化は代替不可の場合のみ

15) 次の工程
- 初期ひな形コードの配置（drivers/db.ts, browser.ts, openai.ts 等）
- config/*.json を作って最初の投稿フローを通す
- 並列ベンチ（4→6→8）で上限を測る

16) Git と Husky の設定（推奨）
- Git 初期化と初回コミット
  git init
  git branch -M main
  git add .
  git commit -m "chore: bootstrap project (v0.4)"
  git remote add origin <YOUR_REPO_URL>
  git push -u origin main

- Husky と lint-staged の導入
  npm i -D husky lint-staged eslint prettier @typescript-eslint/parser @typescript-eslint/eslint-plugin eslint-config-prettier
  npx husky init
  # pre-commit: lint-staged, pre-push: typecheck/build/test を設定（本文のHusky手順参照）

- 動作確認
  1) 適当な変更をして `git add -A`
  2) `git commit -m "test(husky): verify hooks"` → pre-commit で Lint/Prettier が走る
  3) `git push` → pre-push で typecheck/build/test が走る
